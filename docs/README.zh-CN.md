# 亚马逊视觉构型审计 Skill

这是这套 Skill 的中文版长说明，适合国内用户快速理解它的定位、结构、执行方式和复用框架。

## 这套 Skill 在解决什么问题

大多数亚马逊竞品分析会看：

- 价格
- 评论
- 评分
- 关键词
- 标题和五点
- 主图风格

但“主图风格”这个点，很多时候只是停留在经验判断。

比如：

- 这个类目是不是圆形更多？
- 长条形是不是更容易走量？
- 方形是不是偏高客单？
- 高亮材质是不是品类主流？

这些问题如果只靠肉眼看，很容易变成模糊印象，难以复盘，也难以复用。

这套 Skill 的目标，就是把这类模糊判断拆成一个可执行流程：

1. 先采集可见样本
2. 再把产品主图拆成明确标签
3. 再把标签和销量、价格、营收估算做交叉
4. 最后交付一份能讨论的看板

一句话说，它是在回答：

**一个亚马逊品类里，到底哪种产品长相更赚钱？**

## 这套 Skill 为什么适合复用

这套仓库最大的价值，不只是某一次 `shower head` 的分析结果，而是它背后的 Skill 框架。

它有几个很适合复用的点：

### 1. 问题足够窄

不是“做一个亚马逊分析工具”，而是只回答一个问题：

哪种视觉构型和更高的市场表现相关。

问题越窄，流程越稳。

### 2. 每一步都能停下来确认

不是一口气跑完，而是：

- 先确认参数
- 再确认样本
- 再确认打标
- 再确认交叉结果
- 最后交付总报告

这能减少“跑了十几分钟，最后方向错了”的问题。

### 3. 模糊观察被变成明确标签

这一步非常关键。

很多运营工作最大的问题，不是没有观察，而是观察太模糊，没办法重复执行。

这套 Skill 把主图拆成三维标签：

- 几何轮廓
- 结构配置
- 表面工艺

这样后面做聚合和对比时，才不会一直停留在“我感觉这个看起来差不多”。

### 4. 结果不是黑盒结论

最终结论不是凭空来的，而是能回到：

- 哪些产品被采进样本
- 每个产品被打成了什么标签
- 哪些组合贡献了最多营收
- 哪些具体 ASIN 支撑了这个结论

这就是它比一段大提示词更可靠的地方。

## Skill 结构怎么理解

这套 Skill 不是只有一个 `SKILL.md`。

它拆成了几层：

### `SKILL.md`

负责写清楚：

- 目标
- 默认参数
- 边界
- 分步流程
- 停止规则
- 通过条件

你可以把它理解成“这套 Skill 的操作手册”。

### `references/`

负责写节点输入输出和阶段路由。

这层的价值是：让整个流程更像一个清楚的工作流，而不是一段松散说明。

### `scripts/`

负责确定性动作：

- 打标
- 聚合
- 报告生成

适合脚本做的事情，就不要每次交给模型临场发挥。

### `assets/results/`

负责保存阶段产物：

- 样本结果
- 标签结果
- 审计结果
- 最终报告

这能让用户看到每一步做了什么，也方便后续复盘。

## 分步执行应该怎么理解

### 第 0 步：参数确认

这一层先把题目定准。

典型输入：

- 站点：US / UK / DE / JP
- 关键词：例如 `shower head`
- 采样页数：默认前 2 页或前 3 页

这一步的意义不是“收参数”这么简单，而是先锁定分析口径。

### 第 1 步：搜索与样本采集

这一层从 Amazon 搜索页抓样本。

通常会拿到：

- ASIN
- 标题
- 主图链接
- 价格
- 评分
- bought 信号

同时把图片下载下来，为后面的视觉打标做准备。

### 第 2 步：视觉属性打标

这一层开始把商品记录转成视觉样本。

默认做法：

- 轮廓和结构：基于标题关键词判断
- 表面工艺：基于图片亮度、饱和度和方差判断

这一步做完之后，每个商品都不只是“一个商品”，而是一条“带视觉标签的样本”。

### 第 3 步：交叉数据审计

这一层才真正开始回答商业问题：

- 哪种轮廓占营收
- 哪种结构更主流
- 哪种材质感更像品类语言
- 哪些组合兼具销量和价格优势

这里的重点是“交叉”，不是单看某一个字段。

### 第 4 步：总报告交付

最后把前面的结果整理成一份完整看板，适合：

- 团队开会
- 产品讨论
- 设计评审
- 选品预判

## 国内用户怎么复用这套框架

如果你也想做自己的 Codex Skill，可以直接借这套骨架：

### 场景 1：竞品主图结构审计

适合：

- 亚马逊
- 沃尔玛
- 独立站商品集合页

你只需要替换：

- 数据源
- 标签体系
- 最终报告字段

### 场景 2：包装形态研究

把标签从“轮廓/结构/工艺”换成：

- 包装形态
- 颜色体系
- 材料质感
- 开窗方式

### 场景 3：评论 + 视觉联动研究

前面保持视觉打标不变，后面加一层评论主题聚合，就能看：

- 某种视觉构型是否更容易对应某类好评
- 某种结构是否更容易带来某类差评

## 推荐的改造顺序

如果你准备按这套仓库去改自己的 Skill，建议按这个顺序：

1. 先改问题定义，不要先改代码
2. 再改标签体系
3. 再改数据采样范围
4. 最后改报告展示层

因为问题定义和标签体系一旦不稳，后面报告做得再漂亮也没意义。

## 一句话总结

这套仓库不是在教你“怎么写更长的提示词”。

它更像是在示范：

**怎么把一个运营里的模糊判断，沉淀成一个可重复执行、有阶段产物、可复盘的 Codex Skill。**
